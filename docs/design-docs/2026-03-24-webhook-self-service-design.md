## Vision

### 10x Check
The 10x version: users connect GitHub (device flow) and everything else is automatic.
No smee.io URLs, no secrets, no per-repo setup. Webhooks are provisioned transparently
for every workspace. A health dashboard shows real-time event flow. The polling fallback
becomes invisible — users never know it exists because webhooks just work.

## Critical: OAuth Scope Change

The current device flow requests `scope: 'repo'`. Webhook CRUD via the GitHub REST API
requires `admin:repo_hook` (or `write:repo_hook`). The device flow scope must change to
`repo admin:repo_hook`.

**Impact on existing users:** Users who already connected via device flow have a token
with `repo` scope only. They must re-authenticate to gain the webhook management scope.
The UI should detect this (webhook CRUD returns 403 with scope error) and prompt:
"Re-connect GitHub to enable webhook management."

## API Endpoints

All webhook management endpoints are mounted at `/webhooks/manage` (distinct from
the existing `POST /webhooks` receiver in `webhooks.ts`).

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/webhooks/manage/setup` | Generate smee channel + secret, save to config, start smee client |
| `DELETE` | `/webhooks/manage/setup` | Teardown: delete all tracked GH webhooks, clear config, stop smee |
| `GET` | `/webhooks/manage/status` | Health: `{ configured, smeeConnected, lastEventAt, autoProvision, secretPreview }` |
| `POST` | `/webhooks/manage/reload` | Hot-reload: stop old smee client, start new one from fresh config |
| `POST` | `/webhooks/manage/ping` | Send synthetic signed ping event through local `/webhooks` endpoint |
| `POST` | `/webhooks/manage/repos` | Create GitHub webhook for a repo (body: `{ repoPath }`) |
| `POST` | `/webhooks/manage/repos/remove` | Remove GitHub webhook for a repo (body: `{ repoPath }`) |
| `POST` | `/webhooks/manage/backfill` | Create webhooks for all workspaces, returns per-repo results |

**Calling conventions:**
- `POST /webhooks/manage/setup` also starts the smee client (no separate reload needed after initial setup)
- `POST /webhooks/manage/reload` is called by the settings save handler when smeeUrl changes
- `DELETE /webhooks/manage/setup` is the "Remove Setup" button handler — deletes GH webhooks (best-effort), clears webhookSecret/smeeUrl/autoProvision from config, stops smee client
- Repo removal uses POST instead of DELETE to avoid body-stripping by proxies

All endpoints require auth (existing `requireAuth` middleware).

## Key Technical Decisions
- New module: `server/webhook-manager.ts` (25th module — update ARCHITECTURE.md)
- smee channel creation: `GET https://smee.io/new` (follow redirect), NOT `SmeeClient.createChannel()`
- Store webhook IDs in `workspaceSettings[path].webhookId` for deletion tracking
- Per-workspace `webhookEnabled` defaults to global `autoProvision` value at workspace-add time
- Handle 403 gracefully: mark workspace as `webhookError: 'not-admin'`, keep polling for that repo
- Backfill reports per-repo success/failure with inline results
- Hot-reload: `webhook-manager.ts` exports `reloadSmee()`, called by `POST /webhooks/reload`
- Secret never exposed in API responses — only `{ configured: true, secretPreview: '****xxxx' }`
- OAuth scope change: `repo` → `repo admin:repo_hook` (re-auth flow for existing users)
- Auto-cleanup all tracked webhooks before GitHub disconnect (best-effort, log failures)
- smee.io reliability: health indicator distinguishes "relay unreachable" vs "no events" states

## Critical Fix: Unconditional Webhook Receiver Mount

The current `webhooks.ts` router is only mounted if `webhookSecret` exists at startup
(`index.ts:228`). This breaks first-run setup — events would 404 because the receiver
doesn't exist yet. **Fix:** Mount the webhook receiver unconditionally. Validate the
secret at request time (return 401 if no secret configured). This enables the setup
flow to create a secret and immediately start receiving events without restart.

## Codex-Driven Scope Adjustments

1. **Smart polling → M effort** (was S). The current fallback is a blind global timer,
   not per-repo. Making it workspace-aware requires refactoring the broadcast architecture
   to include repo identity. Kept in scope per user decision.
2. **Test/ping → GitHub ping API**. Use `POST /repos/{owner}/{repo}/hooks/{id}/pings`
   for real end-to-end validation (GitHub → smee → server → HMAC → broadcast), not a
   synthetic local POST. Requires at least one workspace with a webhook.
3. **Workspace removal → cleanup webhook**. When a workspace is removed, delete its
   GitHub webhook before removing the workspaceSettings entry. Prevents orphaned hooks.
4. **Config concurrency** — noted risk. read-modify-write on config.json can race between
   setup, backfill, toggle, and disconnect. Low probability (single-user tool) but worth
   serializing config writes via a mutex or queue during batch operations.

## Reviewer Concerns (for /plan-eng-review)

These implementation details were flagged by adversarial spec review and should be
resolved during engineering review:

1. **WebhookDeps interface change** — `webhooks.ts` currently takes `secret: string` in
   `WebhookDeps`. Changing to a getter `() => string | undefined` requires updating the
   interface, the factory call in `index.ts`, and test mocks. Not a "1 line change."
2. **smee.io channel creation fragility** — `GET https://smee.io/new` follows a redirect
   to get the channel URL. This is undocumented behavior with no SLA. Handle non-redirect
   responses (200 with HTML) gracefully.
3. **EventSource lifecycle in webhook-manager** — Track `open`, `error`, `close` events
   on the SSE connection. Handle race conditions during `reloadSmee()` (old client still
   connected while new one starting).
4. **Backfill "shown once"** — Persist via a `backfillOffered: boolean` flag in
   `Config.github`. Re-opening Settings should not re-show the prompt after dismissal.
5. **Manual smee URL input** — Add an "Advanced: Use custom smee URL" expandable section
   in State B for users who self-host. Not a separate UI state.
6. **Secret rotation** — "Remove Setup" + "Setup Webhooks" is the rotation path. No
   dedicated rotate button needed for v1.
7. **Smart polling** — Added to scope: only poll for workspaces where `webhookEnabled`
   is false or `webhookError` is set. Requires refactoring `startWebhookPolling()`.

## Reliability Note

smee.io is a free, community-maintained proxy with no SLA. When smee is unavailable:
1. Health indicator shows "Relay unreachable" (distinct from "no events received")
2. Polling fallback activates automatically (existing 3-error threshold)
3. Manual smee URL input remains available for users who self-host smee

## Eng Review Additions (2026-03-25)

Findings from `/plan-eng-review` that modify the implementation plan:

1. **Extract `extractOwnerRepo()` + `buildRepoMap()` to `server/git.ts`** — currently
   duplicated in `org-dashboard.ts:60` and `review-poller.ts:86`. Both copies deleted,
   `webhook-manager.ts` also imports from the shared location. 4 consumers total.

   `github-app.test.ts`). Update `webhooks.test.ts` for getter pattern change.
